## 2.4 智能化查勘平台开发与数据资产集成

为了将基于 YOLOv11-seg 的红砖墙病害识别能力从算法实验转化为一线查勘可直接使用的工程化工具，课题组围绕“规范采集、自动识别、量化评估、报告输出、资产沉淀”的业务闭环，开发了“红砖墙病害智能检测 Web 系统”。系统采用前后端分离的 B/S 架构，前端基于 Vue3、Element Plus、ECharts 等技术构建交互界面，后端基于 Node.js、Express、Multer 与 Python 推理脚本封装模型服务，并通过本地/服务器端模型权重完成红砖病害的自动识别、结果解析、历史归档与修缮报告生成。平台访问地址为 `http://124.223.57.110:3080`。

该平台并非单纯的算法演示页面，而是围绕历史建筑清水砖墙查勘的真实业务流程进行了系统化集成：一线人员可在浏览器中上传标准化拍摄的墙面照片，系统自动调用后端 YOLOv11 推理引擎，对裂缝、缺损、植物附着、风化、泛碱五类典型病害进行识别，并将模型输出的类别、置信度、空间坐标、病害数量、面积估算、网格位置、严重程度等信息结构化保存。前端进一步将检测结果以标注图、原图叠加框、统计卡片、病害明细、图表看板和修缮报告等形式展示，实现了从单张图像输入到查勘数据资产沉淀的完整闭环。

### 2.4.1 平台总体架构与工程化部署

“红砖墙病害智能检测 Web 系统”采用轻量化、可部署、易维护的 Web 全栈架构，整体分为前端交互层、后端服务层、模型推理层和数据资产层四个部分。

前端交互层主要承担用户操作、图像上传、结果可视化和报告展示功能。系统使用 Vue3 构建单页应用，结合 Element Plus 组件库实现文件上传、参数配置、表格检索、弹窗详情、进度条和标签展示等交互模块；结合 ECharts 和自定义可视化组件实现病害统计、看板模式和分布分析展示。用户无需安装专业软件，只需通过浏览器即可完成现场查勘图像上传、模型识别和报告查看。

后端服务层基于 Express 构建统一 API 服务，主要提供健康检查、图像上传、病害检测、修缮报告生成、用户认证、历史记录查询、记录删除、日志归档等接口。后端通过 Multer 接收前端上传的 JPG、JPEG、PNG 图像，并将图像存储到服务器上传目录，随后调用 Python 推理脚本执行模型识别。系统在接口层对置信度阈值、IoU 阈值、推理图像尺寸等参数进行了范围约束，保证一线人员调参具备一定灵活性的同时，避免异常参数影响推理稳定性。

模型推理层通过后端 `run_inference.py` 脚本封装 YOLOv11 模型推理流程。检测接口会将上传图片路径、置信度阈值、IoU 阈值和推理尺寸传递给 Python 子进程，模型完成识别后返回病害检测列表、图像宽高、类别名称、标注图路径和坐标 TXT 内容。后端再对模型原始输出进行业务化解析，将算法类别编码映射为平台统一显示名称，并生成前端可直接消费的结构化 JSON 数据。

数据资产层则围绕“检测记录—病害结果—报告数据—操作日志”进行沉淀。系统会在每次检测完成后自动写入历史记录，包括记录编号、检测时间、项目名称、图像名称、墙面尺寸、病害类型、病害数量、结果图地址和检测状态等字段。对于平台运行和用户行为，系统同步记录登录、检测、删除等操作日志，为后续项目追溯、质量审计和数据资产复用提供依据。

### 2.4.2 前端查勘工作流与规范化图像输入

平台前端围绕“单图敏捷查勘”进行设计，核心入口位于首页检测模块。用户可以通过拖拽或点击方式上传红砖墙照片，系统限制上传文件类型为 JPG、JPEG 和 PNG，并在界面中提示建议上传清晰的正面照片，以呼应本课题提出的“标准拍摄三步法”。这一设计将前期制定的规范化取证要求嵌入软件操作流程，减少随手拍带来的透视畸变、尺度失真和光照干扰问题。

在正式检测前，前端支持输入或调整参考砖长度参数，默认采用 240mm 作为标准砖长基准。该参数会随图像一并提交至后端，用于辅助完成像素尺度与真实物理尺度之间的换算。与此同时，前端还提供模型推理参数配置能力，包括模型置信度、IoU 阈值和推理图像尺寸等，便于在不同工况下平衡检出率、误报率和推理效率。例如，在裂缝或泛碱等细微病害场景中，可适当提高推理图像尺寸以增强细节捕捉能力；在现场快速筛查场景中，则可采用默认参数完成秒级检测。

用户点击“开始 AI 检测”后，前端将图像文件、参考砖长和推理参数封装为 `FormData`，通过 `/api/detect` 接口提交至后端。检测过程中，界面通过进度条和状态文案提示模型正在分析图片，降低用户等待过程中的不确定性。后端返回结果后，前端自动切换至检测结果区域，显示模型名称、推理模式、置信度阈值、IoU 阈值和推理尺寸等信息，使平台输出具备可解释性和可复核性。

### 2.4.3 YOLOv11 病害识别服务与类别映射机制

后端检测接口是平台智能化能力的核心。系统接收到图像后，首先保存原始上传文件并读取图像宽高信息，随后调用 Python 推理脚本执行 YOLOv11 模型识别。推理脚本返回的结果包含每个病害目标的类别编号、类别名称、置信度、边界框坐标、标注图路径以及坐标文件内容。后端进一步通过统一的类别映射表将模型类别编码转换为工程业务名称：`01:LF` 对应裂缝，`02:QS` 对应缺损，`03:P` 对应植物附着，`04:B-FH` 对应风化，`05:B-FJ` 对应泛碱。该映射机制保证了模型训练标签、后端业务数据和前端展示名称之间的一致性。

在结果解析过程中，系统会过滤不属于五类目标病害的无效类别，并为每个有效检测目标生成唯一编号、病害类型、原始类别名、置信度、边界框、面积估算值、严重程度和网格编号。对于风化和泛碱两类具有明显面状分布特征、与修缮面积关系更直接的病害，系统根据参考砖长和图像像素尺度估算其受损面积，单位为平方米。对于裂缝、植物附着和缺损等病害，系统更多以处数、位置和严重程度作为修缮管理依据。

系统还依据置信度对病害严重程度进行初步分级：高置信度目标标记为重度，中等置信度目标标记为中度，其他有效目标标记为轻度。虽然该分级不能替代专家最终诊断，但可以在现场快速查勘阶段提供修缮优先级参考。与此同时，系统将图像划分为规则网格，并根据目标边界框所在位置生成类似 `G1-1`、`G2-3` 的网格编号，为后续墙面分区管理、病害热力图、QGIS 关联分析和空间资产化管理提供基础字段。

### 2.4.4 检测结果可视化与数据看板联动

检测完成后，前端结果区从图像、统计、明细和看板四个层面展示模型输出。图像展示部分支持“标注图”和“原图+框”两种视图：标注图由模型推理脚本自动生成，直接显示类别代码和置信度；原图叠加框则由前端根据后端返回的边界框坐标动态绘制，能够在浏览器中按照图像缩放比例实时对齐病害位置。

为了提升多类别病害的识别辨识度，平台为五类病害设置了统一颜色体系：风化为红色，泛碱为蓝色，裂缝为橙色，植物附着为紫色，缺损为青绿色。该颜色体系在前端图例、边界框、病害列表和报告组件中保持一致，使检测结果具有较高的可读性和专业展示效果。

统计展示部分以卡片形式呈现检测到的病害总数、病害种类数量、风化面积和泛碱面积等核心指标。病害明细区按照类别汇总每类病害的检出数量、估算面积和最高严重程度，并通过进度条展示其在全部检测目标中的占比。对于管理人员而言，该界面能够快速回答“这张墙面照片中有哪些病害、各有多少处、哪些病害面积较大、哪些病害风险较高”等关键问题。

平台还提供看板模式，将图像检测结果与统计图表进行联动展示。该模式面向工程管理和汇报场景，能够更直观地呈现单张照片中的病害结构、类别分布和空间位置关系，为现场查勘后的讨论、复核和方案制定提供可视化支撑。

### 2.4.5 修缮报告自动生成与工程量辅助评估

在检测结果基础上，用户可一键生成修缮报告。前端调用 `/api/report/generate` 接口，将完整检测结果提交给后端，由后端按照预设规则生成结构化报告数据。报告内容包括报告标题、生成时间、检测时间、原始图像路径、参考砖长、专业说明、总体评估、网格分析、病害详情和修缮优先级计划等。

总体评估模块会统计病害类型数量、病害总数、风化面积、泛碱面积和综合风险等级。系统根据病害数量和病害种类对墙面状态进行初步风险判断：当检测目标数量较多或病害类型较复杂时，标记为高风险；当病害数量和类型处于中间水平时，标记为中风险；其他情况则标记为低风险。报告同步生成相应的处置建议，如立即开展修缮、尽快制定修缮计划或纳入定期巡检。

病害详情模块针对风化、泛碱、裂缝、植物附着和缺损五类病害分别展示是否检出、检出数量、总面积、最高严重程度、病害描述、严重程度说明、修缮方法、修缮材料和预防措施。该模块将模型识别结果与工程修缮知识进行绑定，使平台输出从“算法识别结果”进一步转化为“可用于工程决策的诊断建议”。

网格分析模块按照病害目标所在网格统计每个分区的病害数量、病害类型、最高严重程度和优先级。该设计为后续将检测结果接入 QGIS、建立墙面病害空间索引、生成病害热力图和开展分区修缮计划提供了数据基础。报告组件还支持导出 JSON、Markdown、Word 类文本和打印等形式，其中 JSON 结构面向后续 GIS 或数据平台集成，Markdown 与打印功能则便于结题报告、现场记录和修缮归档使用。

### 2.4.6 历史记录、用户认证与数据资产归档

为了避免检测结果停留在一次性页面展示层面，平台实现了历史记录归档机制。每次检测完成后，后端会自动调用历史记录写入函数，生成一条检测资产记录。记录字段包括唯一 ID、记录编号、检测时间、检测状态、项目名称、图像名称、墙面尺寸、病害类型数组、病害数量、结果图 URL 和报告 URL 等。通过这些字段，单次检测被转化为可检索、可追溯、可复用的数据资产。

前端历史记录页面支持按照项目名称、病害类型、检测状态和时间范围进行筛选，并支持分页查询。用户可以查看历史检测详情、删除无效记录或触发报告导出流程。对于长期巡检任务而言，该功能能够帮助管理人员持续积累同一建筑、同一墙面或同一项目的病害演变数据，从而支撑预防性保护和全生命周期维护。

平台同时实现了用户注册、登录、退出和用户信息查询等基础认证能力。后端通过认证中间件保护历史记录、日志等管理接口，保证检测数据和项目数据不会被未授权访问。系统还记录用户登录、退出、检测、删除等关键操作日志，日志内容包括用户名、操作类型、操作描述、IP 地址、状态、请求地址和请求方法等，为后续数据审计、责任追踪和企业级系统集成奠定基础。

该数据归档方式使平台具备从“工具软件”向“数据资产系统”演进的条件。随着项目持续运行，系统可不断沉淀不同建筑、不同光照、不同病害类型和不同修缮阶段的图像及识别结果，为后续模型增量训练、难样本回流、病害时序分析和修缮知识库建设提供真实工程数据来源。

### 2.4.7 数据资产集成与模型持续优化闭环

平台的数据资产集成主要体现在三个方面：一是原始影像资产的集中化管理，二是模型识别结果的结构化表达，三是报告和历史记录的业务化归档。原始上传图像和模型标注图被统一保存于服务器静态资源目录，前端通过 URL 调用展示；识别结果则以 JSON 形式在前后端之间传递，包含病害类别、置信度、坐标、面积、严重程度、网格编号和汇总统计等关键字段；历史记录和操作日志进一步记录检测时间、项目来源、图像名称和处理状态，使每一次查勘都能够形成完整的数据链条。

从数据格式看，平台输出已经具备与外部系统对接的基础。检测结果中的边界框坐标、网格编号、病害类别和面积字段可用于 GIS 空间标注；报告导出的 JSON 数据可作为后续 QGIS、数字孪生平台或企业资产管理平台的数据交换格式；历史记录中的项目名称、检测时间和病害类型可用于构建多项目、多批次的病害统计台账。

从模型优化角度看，平台为主动学习和难样本回流提供了工程入口。当系统在真实场景中遇到低置信度、强反光、脚手架遮挡、植物覆盖、红砖斑驳纹理等复杂样本时，可将其从历史记录和上传图像中筛选出来，回流至人工复核和二次标注环节。复核后的样本可重新并入训练集，推动 YOLOv11 模型进行周期性增量训练，从而逐步提升模型对真实历史建筑复杂工况的泛化能力。

因此，平台不仅完成了模型推理服务的封装，也将数据采集、模型识别、结果展示、报告生成、历史归档和质量审计连接成闭环。该闭环为本课题后续建设“红砖建筑病害数据资产库”和“智能化查勘持续优化体系”提供了可落地的软件基础。
